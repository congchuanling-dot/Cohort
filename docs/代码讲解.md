# foundation 包

## Message — 消息体

> 源文件：[internal/foundation/message.go](../internal/foundation/message.go)

### 概述

`Message` 是智能体之间的通信单元，也是整个框架的数据载体。

### 核心路由字段

两个字段控制消息的流转：

| 字段 | 作用 | 路由阶段 |
|------|------|----------|
| `CauseBy` | 标识消息由哪个 Action 产生 | Role 订阅（watch）的依据 |
| `SendTo` | 标识消息发送给哪些 Role | Environment 路由的依据 |

### 完整结构

```go
type Message struct {
    ID              string         // 唯一标识，格式: msg_<原子递增>
    Content         string         // 自然语言内容
    InstructContent any            // 结构化数据（PRD、代码等）
    Role            string         // user / system / assistant
    CauseBy         string         // 由哪个 Action 产生
    SentFrom        string         // 由哪个 Role 发送
    SendTo          []string       // 接收者列表
    Metadata        map[string]any // 扩展元数据
    Timestamp       time.Time      // 创建时间
}
```

### 特殊路由常量

```go
RouteToAll  = "<all>"   // 广播给所有角色
RouteToSelf = "<self>"  // 回环给自己
```

### 工厂方法

- `NewUserMessage(content)` — 创建用户消息，`CauseBy="UserRequirement"`，广播给所有 Role
- `NewSystemMessage(content, causedBy, sentFrom)` — 由 Agent 产生的系统消息

### 路由判断方法

**ShouldSendTo** — 遍历 `SendTo` 列表，若任一目标为 `<all>`、传入的 `roleName` 或 `"*"`，则返回 `true`。

**ShouldObserve**：
- `watchKeys` 为空 → 关注所有消息
- `watchKeys` 非空 → 按 `CauseBy` 查 map，存在则关注

**设计意图**：`SendTo` 是推送路由（Environment 决定发给谁），`ShouldObserve` 是拉取过滤（Role 决定关注什么）。

### 全局 ID 生成

```go
var idCounter atomic.Int64  // 原子计数器，零外部依赖

func init() {
    idCounter.Store(time.Now().UnixNano())  // 从当前纳秒时间戳起步
}
```

用 `atomic.Int64` 而非 `uuid` 库，保持零外部依赖。从 `UnixNano()` 起步而非 0，避免短会话内 ID 碰撞。

## Memory — 消息存储器

> 源文件：[internal/foundation/memory.go](../internal/foundation/memory.go)

### 概述

`Memory` 是框架的消息存储器，核心职责是**存储 Message 并提供高效检索**。

### 数据结构

```go
type Memory struct {
    mu      sync.RWMutex          // 读写锁：读操作不互斥，写操作独占
    storage []*Message            // 有序消息列表（按插入顺序）
    index   map[string][]*Message // cause_by → 消息列表的倒排索引
    maxSize int                   // 容量上限，0 = 无限制
}
```

**双存储设计**：`storage` 保证时序，`index` 保证按 `cause_by` 的 O(1) 查找。

### 方法分类

| 类别 | 方法 | 复杂度 | 说明 |
|------|------|--------|------|
| **写入** | `Add` | O(1) | 追加 + 索引 + FIFO 淘汰 |
| **全量读取** | `Get(k)` | O(k) | 取最近 k 条 |
| | `Last` | O(1) | 取最后一条 |
| **索引读取** | `GetByAction` | O(1) | 按 cause_by 查 |
| | `GetByActions` | O(m) | 多 cause_by OR 查询 |
| **线性扫描** | `GetByRole` | O(n) | 按发送者遍历 |
| | `FindNews` | O(n) | 找未读消息 |
| **维护** | `Count` / `Clear` | O(1) / O(n) | |

### 关键设计点

**1. FIFO 淘汰（Add 方法）**

当 `len(storage) > maxSize` 时淘汰最旧消息。同时调用 `removeFromIndex` 保持索引和存储一致——否则索引中残留已删除消息的指针（**内存泄漏 + 脏数据**）。

**2. 为什么 GetByRole 用线性扫描而没有索引？**

`SentFrom`（发送者）基数通常很小（一个系统就几个 Role），建索引收益不大，反而增加写入开销和内存。`cause_by` 不同——每个 Action 都产生消息，种类多，索引价值高。

**3. 并发安全**

所有公开方法都用 `RLock`（读）或 `Lock`（写）保护。读写锁的收益：读取频率远高于写入时，多个 goroutine 可同时读而不互斥。

### 与 Message 的协作

```
Memory.storage → []*Message
                      ├── CauseBy  → Memory.index 的 key
                      ├── SentFrom → GetByRole 的过滤条件
                      ├── SendTo   → ShouldSendTo 的路由判断
                      └── ID       → FindNews 的去重依据
```

`Memory` 是"仓库"，`Message` 是"货物"，两者共同支撑起 `Environment` 的消息路由和 `Role` 的订阅机制。

### 消息已读/未读机制

> 涉及文件：[memory.go](../internal/foundation/memory.go) + [role.go](../internal/role/role.go)

**核心思想：不在 Message 上加 `Read` 字段，而是每个 Role 各自记账。**

```
Role.observed []string          ← 每个 Role 私有的已读 ID 列表
        ↓
markObserved(msg)               ← 收到消息就追加 ID
        ↓
Memory.FindNews(observed, k)    ← 差集运算：storage 中不在 observed 里的 = 未读
```

**Step 1 — Role 收到消息后标记已读（role.go:217-220）**

```go
r.memory.Add(msg)       // 写入 Memory 仓库
r.markObserved(msg)     // 记录 ID 到自己的 observed 列表
```

消息一旦通过 `shouldObserve` 过滤并被 Role 处理，就算"已读"。

**Step 2 — markObserved 追加 ID，硬上限 1000（role.go:417-422）**

```go
func (r *Role) markObserved(msg *foundation.Message) {
    r.observed = append(r.observed, msg.ID)
    if len(r.observed) > 1000 {
        r.observed = r.observed[100:] // 丢弃最旧的 100 条，保留最近 900 条
    }
}
```

1000 条是硬上限，防止长期运行的 Role 内存无限增长。超过时丢弃最旧的 100 条（意味着这些"已读"记录会被遗忘，对应的消息会重新变成"未读"——但通常这些消息早已被淘汰或不再需要）。

**Step 3 — FindNews 做差集（memory.go:111-123）**

```go
func (m *Memory) FindNews(observed []string, k int) []*Message {
    observedSet := make(map[string]bool, len(observed))  // 转 set，O(1) 查重
    for _, id := range observed {
        observedSet[id] = true
    }
    var news []*Message
    for i := len(m.storage) - 1; i >= 0 && len(news) < k; i-- {  // 从最新往前扫
        if !observedSet[m.storage[i].ID] {  // 不在已读集合 = 未读
            news = append(news, m.storage[i])
        }
    }
    return news
}
```

**设计优点：**

| 维度 | 说明 |
|------|------|
| **无状态污染** | Message 是纯数据对象，不嵌入运行时状态 |
| **多 Role 解耦** | 每个 Role 各自维护 `observed`，A 读了不代表 B 也读 |
| **无并发竞争** | `observed` 是 Role 私有字段，只在主循环 goroutine 中读写 |
| **低成本** | 差集运算只在调用 `FindNews` 时才发生，不是每条消息都做 |

**与"推送 vs 拉取"的关系：**

- `shouldObserve(watchKeys)` → **推送端过滤**：Environment 把消息投递到 msgBuffer 前，先判断该 Role 是否关注此 cause_by
- `FindNews(observed)` → **拉取端查询**：Role 主动查询"我还没看过哪些消息"
- `markObserved` → **状态标记**：处理完一条就记录一条

三者配合，形成完整的消息投递→过滤→标记→查询闭环。

# tool 包

## Tool 接口 + ToolRegistry

> 源文件：[internal/tool/tool.go](../internal/tool/tool.go)

### 概述

`tool.go` 定义了 Agent 框架的**工具层**——不依赖 LLM 的基础操作能力。如果把 Agent 比作一个人，Tool 是"手"（执行确定性操作），Action 是"脑"（LLM 驱动的决策）。

### Tool 与 Action 的分工

| | Tool | Action |
|---|---|---|
| 依赖 LLM？ | ❌ 不依赖 | ✅ AskLLM 是核心方法 |
| 本质 | 纯函数 | LLM 驱动的决策流程 |
| 示例 | ReadFile、WriteFile、RunCommand | WriteCode、ReviewCode |

**为什么要分开？** 如果文件读写和代码生成混在一起，LLM 调用会变得不可控。Tool 提供确定性的底层能力，Action 在此基础上做智能决策。

### 公有/私有分层

- **公有 Tool** 注册在 Environment 级别 → 所有 Role 共享（如 ReadFile，谁都需要）
- **私有 Tool** 注册在 Role 级别 → 只有该 Role 能用（如某个 Role 专属的外部 API）

### Tool 接口

```go
type Tool interface {
    Name() string                          // 唯一标识
    Description() string                   // 给 LLM 看的说明
    Parameters() map[string]string         // 参数说明
    Run(ctx context.Context, args map[string]any) (string, error)  // 执行
}
```

**设计核心**：`Description()` 和 `Parameters()` 的输出会注入 LLM 的 system prompt，让 LLM 知道有哪些工具可用以及如何调用。这意味着**工具描述即 prompt 工程**——描述写得好不好直接影响 LLM 调用工具的准确率。

### ToolRegistry 注册表

```go
type ToolRegistry struct {
    mu    sync.RWMutex       // 读写锁：读不互斥，写独占
    tools map[string]Tool    // name → Tool
}
```

| 方法 | 复杂度 | 说明 |
|------|--------|------|
| `Register(t)` | O(1) | 注册工具，重名 panic（视为编程错误） |
| `RegisterAll(tools...)` | O(n) | 批量注册 |
| `Get(name)` | O(1) | 按名查询，不存在返回 nil |
| `Call(ctx, name, args)` | O(1) | 查询 + 执行 |
| `List()` | O(n log n) | 按字母排序返回名称列表 |
| `Count()` | O(1) | 返回数量 |
| `Merge(other)` | O(n) | 合并另一个注册表，重名不覆盖 |
| `ToolsInfo()` | O(n log n) | 生成 LLM system prompt 用的工具描述文本 |

**关键设计点：**

1. **重名用 panic 而非 error** — 工具重名是编程错误，应在开发阶段暴露，panic 强制修复
2. **Merge 重名不覆盖** — 私有工具优先级高于公有工具。如果 Role 注册了自己的 `ReadFile`（加了权限检查），Environment 的通用版本不会覆盖它
3. **ToolsInfo 排序输出** — 保证每次生成的 prompt 顺序一致，对 LLM prompt cache 友好
4. **空注册表返回空字符串** — 不出现空 "Available Tools" 段落，避免 LLM 困惑

### Merge 的设计意图

```
Environment.ToolRegistry (公有) ──Merge──→ Role.ToolRegistry (私有)
                                              ├── 已有私有工具（保留）
                                              └── 公有工具（追加，不覆盖同名私有工具）
```

### System Prompt 生成

`ToolsInfo()` 输出格式：
```
## Available Tools
1. **ReadFile**: 读取指定路径的文件内容
   Parameters: path (文件路径)
2. **WriteFile**: 将内容写入文件
   Parameters: path (文件路径), content (文件内容)
```

### 与 LLM 的协作流程

```
初始化阶段:
  注册所有 Tool → ToolsInfo() 生成描述 → 注入 LLM system prompt

运行阶段:
  LLM 输出"调用 ReadFile path=xxx" → Call(ctx, "ReadFile", args) → Tool 执行 → 结果返回 LLM
```

### 整体架构图

```
┌─────────────────────────────────────────────┐
│                 Environment                  │
│  ┌───────────────────────────────────────┐  │
│  │       ToolRegistry (公有)              │  │
│  │  ReadFile │ WriteFile │ RunCommand    │  │
│  └───────────────────────────────────────┘  │
│                    │                         │
│              Merge 注入                      │
│         ┌─────────┼─────────┐               │
│         ▼         ▼         ▼               │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐    │
│  │  Role A  │ │  Role B  │ │  Role C  │    │
│  │ 私有工具  │ │ 私有工具  │ │ 私有工具  │    │
│  │+公有工具  │ │+公有工具  │ │+公有工具  │    │
│  └──────────┘ └──────────┘ └──────────┘    │
│                                              │
│  ToolsInfo() → 注入 LLM System Prompt        │
│  LLM 看到可用工具列表 → 决策调用哪个工具       │
└─────────────────────────────────────────────┘
```

### 附带 Go 语法讲解

| 语法 | 位置 | 说明 |
|------|------|------|
| `type Tool interface{}` | 第 29 行 | 接口定义，Go 是隐式实现，不需要 `implements` |
| `sync.RWMutex` | 第 54 行 | 读写锁，读操作不互斥、写操作独占。读多写少场景用 |

| `tools ...Tool` | 第 76 行 | 可变参数，调用方可传 0~N 个 |
| `strings.Builder` | 第 155 行 | 高效字符串拼接，内部用 `[]byte` 缓冲区 |
| `sort.Strings()` | 第 106 行 | 标准库排序，对字符串切片原地排序 |
| `context.Context` | 第 42 行 | 传递超时/取消/截止时间信号 |

### 与 builtin 工具的关系

`tool.go` 只定义接口和注册表，具体实现在 `internal/tool/builtin/` 目录：

| 文件 | 工具 | 说明 |
|------|------|------|
| [read_file.go](../internal/tool/builtin/read_file.go) | ReadFileTool | 读文件，限制输出 10000 字符 |
| [write_file.go](../internal/tool/builtin/write_file.go) | WriteFileTool | 写文件 |

每个 builtin 工具都是一个空 struct（如 `type ReadFileTool struct{}`），实现 `Tool` 接口的 4 个方法。空 struct 不占用内存，是 Go 中"无状态服务"的惯用写法。


## WriteFile — 文件写入工具

> 源文件：[internal/tool/builtin/write_file.go](../internal/tool/builtin/write_file.go)

### ReadFile vs WriteFile 设计差异

| 维度 | ReadFile | WriteFile |
|------|----------|-----------|
| 结果方向 | 文件 → 内存（有截断风险） | 内存 → 文件（天然安全） |
| 父目录 | 不关心，不存在就报错 | **自动创建** `MkdirAll` |
| 输出信息 | 返回文件内容文本 | 返回"✅ 已写入 xxx（N 字节）" |
| content 缺失 | N/A | 不报错，写空文件（视为合法操作） |

两者的核心不对称：**读是消费，写是生产**。读要防数据过载，写要帮铺路——WriteFile 独有的 `MkdirAll` 意味着写一个深层路径不需要先建目录。

### Run 方法三段

**1. 参数提取（宽松策略）**

```go
path, _ := args["path"].(string)
if path == "" {
    return "", fmt.Errorf("WriteFile: 'path' 参数缺失或不是字符串")
}
content, _ := args["content"].(string)  // 不判 ok，content 为空也算合法
```

`path == ""` 一条判断同时覆盖 key 不存在和类型不对——断言失败后值变为零值 `""`，同样被捕获。content 不判空是因为**写空文件是合法操作**（清空文件）。

**2. 自动建父目录**

```go
dir := filepath.Dir(path)        // filepath.Dir("a/b/c.go") → "a/b"
os.MkdirAll(dir, 0755)           // 等同于 mkdir -p
```

`MkdirAll` 递归创建不存在的父目录。权限 `0755` = `rwxr-xr-x`——目录必须有执行位才能 `cd` 进去。

**3. 写入**

```go
os.WriteFile(path, []byte(content), 0644)
return fmt.Sprintf("✅ 已写入 %s（%d 字节）", path, len(content))
```

返回值三个信息：✅ 状态 + 路径 + 字节数，LLM 看到这个就能判断是否成功。`0644` = 文件权限 `rw-r--r--`。


## ReadFile — 文件读取工具

> 源文件：[internal/tool/builtin/read_file.go](../internal/tool/builtin/read_file.go)

### Run 方法四段

**1. 参数校验（类型断言 comma ok 模式）**

```go
path, ok := args["path"].(string)
if !ok || path == "" {
    return "", fmt.Errorf("ReadFile: 'path' 参数缺失或不是字符串")
}
```

`comma ok` 是防御性编程——类型断言不匹配时不会 panic，而是 `ok=false`。

**2. 执行读取（错误包装 `%w`）**

```go
data, err := os.ReadFile(path)  // Go 1.16+ 便捷函数
return fmt.Errorf("ReadFile: 读取 %s 失败: %w", path, err)  // %w 保留原始错误链
```

`%w` 是 Go 1.13 的**错误包装**动词，调用方能用 `errors.Is()` 解包判断原始错误类型。

**3. 截断保护**

```go
if len(data) > 10000 {
    data = data[:10000]
    return fmt.Sprintf("...（文件共 %d 字节）", len(data)+10000)
}
```

10000 字符 ≈ 2500-4000 tokens，防止撑爆 LLM 上下文窗口。

**已知 Bug**：`data[:10000]` 截断后 `len(data)` 变成 10000，所以 `len(data)+10000` 永远 = 20000，实际文件大小信息丢失。应改为先保存 `origLen := len(data)` 再截断。

更深层的问题见 [待改进文档](待改进-ReadFile截断问题.md)：截断是"替 LLM 做决策"，理想做法是让 LLM 判断保留什么——但这应在 Action 层实现（Action 内先调 Tool 拿数据，再调 AskLLM 做摘要）。

# action 包

## Action 接口 + BaseAction

> 源文件：[internal/action/action.go](../internal/action/action.go)

### Action 接口（2 个方法）

```go
type Action interface {
    Name() string   // 名称 = Message.CauseBy（消息路由的依据）
    Run(ctx context.Context, history []*foundation.Message) (*ActionOutput, error)
}
```

**与 Tool 接口对比**：

| | Tool | Action |
|---|---|---|
| 方法数 | 4（Name/Desc/Params/Run） | 2（Name/Run） |
| 输入参数 | `map[string]any` | `[]*foundation.Message` |
| 为什么少？ | 描述和参数给 LLM 看 | Action 不需要——LLM 不直接调 Action |

### Name() 的双重身份

```
Action.Name() ←→ Message.CauseBy
```

WritePRD 的 `Name()` 返回 `"WritePRD"`，执行后产出的 Message 的 `CauseBy` 也是 `"WritePRD"`。下一个 Action（WriteDesign）通过 `findLatestByAction(history, "WritePRD")` 找到它——Action 之间通过 Message history **间接通信**，不直接互相调用。

### ActionOutput

```go
type ActionOutput struct {
    Content         string  // 自然语言输出（给其他 Agent 读的）
    InstructContent any     // 结构化输出（给下一个 Action 用的，如 PRD 字段）
}
```

同一个产出的两种表达——`Content` 给人看，`InstructContent` 给程序用。

### BaseAction 结构体（5 个字段）

```go
type BaseAction struct {
    name   string            // Action 名称 = Message.CauseBy
    prefix string            // System Prompt（给 LLM 的"人设"）
    client llm.Client        // LLM 客户端
    node   *ActionNode       // 结构化输出解析器（可选）
    tools  *tool.ToolRegistry // 工具注册表（由 Role 注入）
}
```

所有具体 Action（WritePRD、WriteCode 等）通过**嵌入结构体**继承 BaseAction，只实现自己的 `Run` 方法。

### AskLLM —— 核心方法（4 步）

```go
func (a *BaseAction) AskLLM(ctx context.Context, prompt string,
    history []*foundation.Message) (string, error) {

    // Step 1: 构建 system prompt（prefix + 工具列表）
    systemPrompt := a.prefix + a.tools.ToolsInfo()

    // Step 2: 历史消息翻译（framework Message → LLM ChatMessage）
    historyMsgs := a.frameToLLMMessages(history)

    // Step 3: 附加当前 prompt
    messages = append(messages, ChatMessage{Role: "user", Content: prompt})

    // Step 4: 调 LLM
    resp, err := a.client.Chat(ctx, messages)
}
```

最终发给 LLM 的消息序列：

```
[system]  You are a professional Product Manager...  ← prefix
          ## Available Tools                          ← toolsInfo
[user]    <历史消息 1>                                ← history
[user]    <历史消息 2>
[user]    Please write a detailed PRD based on...     ← prompt（★ Action 传进来的）
```

### frameToLLMMessages —— 格式转换

```go
func (a *BaseAction) frameToLLMMessages(msgs []*foundation.Message) []llm.ChatMessage {
    for _, msg := range msgs {
        role := msg.Role
        if role == "" { role = "user" }  // 空 Role 兜底
        result = append(result, ChatMessage{Role: role, Content: msg.Content})
    }
}
```

框架的 `Message`（有 ID/CauseBy/SendTo 等路由字段）→ LLM 的 `ChatMessage`（只有 Role+Content）。路由信息全丢，LLM 只需要对话内容。

### 完整调用链

```
Role.reactByOrder()
  ├─ injectToolsToAction(act)        ← 注入工具
  ├─ act.Run(ctx, history)            ← 调用 Action
  │    └─ 从 history 中翻上游产出
  │    └─ 拼 prompt
  │    └─ a.AskLLM(ctx, prompt, history)
  │         ├─ prefix + toolsInfo → system prompt
  │         ├─ history → LLM 格式
  │         ├─ prompt → user 消息
  │         └─ client.Chat() → 真调 LLM
  └─ 返回 Message{CauseBy: act.Name()}
```

### 工具注入链路

```
Environment.RegisterRole → r.MergePublicTools(public)
Role.reactByOrder()       → r.injectToolsToAction(act)
                            → act.SetTools(r.tools)
Action.Run()              → a.CallTool(ctx, "WriteFile", args)
```

三个入口——公有工具注册在 Environment，私有工具注册在 Role，执行时由 Role 注入到 Action。延迟绑定，Action 不知道自己会被分配给哪个 Role。

### Run 主循环 —— observe → think → act → publish

```go
func (r *Role) Run(ctx context.Context) error {
    for {                                    // 无限循环
        select {                             // 同时等两个 channel
        case <-r.ctx.Done():                 // cancel 信号 → 退出
            return r.ctx.Err()
        case msg := <-r.msgBuffer:           // 新消息 → 处理
            // Step 1: Observe
            if !r.shouldObserve(msg) { continue }
            r.memory.Add(msg)
            r.markObserved(msg)

            // Step 2+3: Think + Act（合并为 react）
            rsp, _ := r.react(r.ctx)

            // Step 4: Publish
            if rsp != nil { r.env.PublishMessage(rsp) }
        }
    }
}
```

**`context.WithoutCancel`**（Go 1.21）：env 的 cancel 用于控制主循环启停，但 LLM 调用一旦发起必须完成——用 `WithoutCancel` 让 react 内部的 LLM 调用不被 env cancel 中断。

## SOP 模式（reactByOrder）

### 核心：state 指针推进

```
actions = [WritePRD, WriteDesign, WriteCode]
              ↑ state=0：下一个执行 WritePRD
                        ↑ state=1：下一个执行 WriteDesign
                                    ↑ state=2：下一个执行 WriteCode
                                                ↑ state=3 → state=-1 → idle
```

### 逐行拆解

```go
func (r *Role) reactByOrder(ctx context.Context) (*foundation.Message, error) {
    // ① 活干完了？下班
    if r.state >= len(r.actions) {
        r.state = -1
        return nil, nil
    }

    // ② 取出当前 Action + 拿全部历史
    act := r.actions[r.state]
    history := r.memory.Get(0)   // Get(0) = 全部

    // ③ 注入工具（每次执行前注入，因为工具可能变化）
    r.injectToolsToAction(act)

    // ④ 干活（内部调 AskLLM）
    output, err := act.Run(ctx, history)
    if err != nil { return nil, err }

    // ⑤ 推进指针
    r.state++
    if r.state >= len(r.actions) { r.state = -1 }

    // ⑥ 包装成 Message 返回
    return &foundation.Message{
        Content:  output.Content,
        CauseBy:  act.Name(),          // ★ Name = CauseBy（下游靠这个订阅）
        SentFrom: r.Name,
        SendTo:   []string{RouteToAll},
        Role:     "system",
    }, nil
}
```

### state 运行实例

```
Alice: actions=[WritePRD]  watch=[UserRequirement]

第1轮：收到 UserRequirement → state=0 → 执行 WritePRD → state++ → state=1 → state=-1
       发布 Message{CauseBy:"WritePRD"}

第2轮：收到任何消息 → state=-1 >= 1 → return nil（idle，不干活）

Bob:   actions=[WriteDesign] watch=[WritePRD]

第1轮：收到 UserRequirement → shouldObserve? watch=[WritePRD] ≠ "UserRequirement" → skip
第2轮：收到 Message{CauseBy:"WritePRD"} → match → state=0 → 执行 WriteDesign → state=-1
       发布 Message{CauseBy:"WriteDesign"}
```

### SOP 完整链路

```
用户输入 "写一个 Todo 应用"
  → Team.Run() → PublishMessage{CauseBy:"UserRequirement"}
    → Alice 收到 → WritePRD.Run() → AskLLM → 返回 PRD
      → PublishMessage{CauseBy:"WritePRD"}
        → Bob 收到 → WriteDesign.Run() → AskLLM → 返回技术设计
          → PublishMessage{CauseBy:"WriteDesign"}
            → Charlie 收到 → WriteCode.Run() → AskLLM → 返回代码
              → PublishMessage{CauseBy:"WriteCode"}
                → 所有 Role idle → 结束
```

## Go 语法专题：select + channel + for

### for{} — 死循环

```go
for { }  // Go 的 while true
```

### select — 同时等多个 channel

```go
select {
case a := <-chA:   // chA 有数据 → 走这
case b := <-chB:   // chB 有数据 → 走这
}
```

类比等电话和门铃——哪个先响处理哪个。不是轮询，Go 运行时保证高效等待。

### channel 方向

```go
ch <- value    // 往 channel 发（箭头指向 channel）
value := <-ch  // 从 channel 取（箭头从 channel 出来）
```

### 取值的 ok 标志

```go
msg, ok := <-ch
// ok=true  → 取到了
// ok=false → channel 关了
```

### 完整翻译

```go
for {                               // 一直循环
    select {                        // 同时等两件事
    case <-ctx.Done():              // 情况 A：context 说"停"
        return                      //   退出
    case msg := <-msgBuffer:        // 情况 B：邮箱有信
        // 处理信                    //   拆信干活
    }
}
```

### struct{} 空结构体

```go
type ReadFileTool struct{}  // 不占内存，纯方法载体
```

Go 的零内存类型——当 struct 只是方法的载体而不需要存储任何状态时用。
