# foundation 包

## Message — 消息体

> 源文件：[internal/foundation/message.go](../internal/foundation/message.go)

### 概述

`Message` 是智能体之间的通信单元，也是整个框架的数据载体。

### 核心路由字段

两个字段控制消息的流转：

| 字段 | 作用 | 路由阶段 |
|------|------|----------|
| `CauseBy` | 标识消息由哪个 Action 产生 | Role 订阅（watch）的依据 |
| `SendTo` | 标识消息发送给哪些 Role | Environment 路由的依据 |

### 完整结构

```go
type Message struct {
    ID              string         // 唯一标识，格式: msg_<原子递增>
    Content         string         // 自然语言内容
    InstructContent any            // 结构化数据（PRD、代码等）
    Role            string         // user / system / assistant
    CauseBy         string         // 由哪个 Action 产生
    SentFrom        string         // 由哪个 Role 发送
    SendTo          []string       // 接收者列表
    Metadata        map[string]any // 扩展元数据
    Timestamp       time.Time      // 创建时间
}
```

### 特殊路由常量

```go
RouteToAll  = "<all>"   // 广播给所有角色
RouteToSelf = "<self>"  // 回环给自己
```

### 工厂方法

- `NewUserMessage(content)` — 创建用户消息，`CauseBy="UserRequirement"`，广播给所有 Role
- `NewSystemMessage(content, causedBy, sentFrom)` — 由 Agent 产生的系统消息

### 路由判断方法

**ShouldSendTo** — 遍历 `SendTo` 列表，若任一目标为 `<all>`、传入的 `roleName` 或 `"*"`，则返回 `true`。

**ShouldObserve**：
- `watchKeys` 为空 → 关注所有消息
- `watchKeys` 非空 → 按 `CauseBy` 查 map，存在则关注

**设计意图**：`SendTo` 是推送路由（Environment 决定发给谁），`ShouldObserve` 是拉取过滤（Role 决定关注什么）。

### 全局 ID 生成

```go
var idCounter atomic.Int64  // 原子计数器，零外部依赖

func init() {
    idCounter.Store(time.Now().UnixNano())  // 从当前纳秒时间戳起步
}
```

用 `atomic.Int64` 而非 `uuid` 库，保持零外部依赖。从 `UnixNano()` 起步而非 0，避免短会话内 ID 碰撞。

## Memory — 消息存储器

> 源文件：[internal/foundation/memory.go](../internal/foundation/memory.go)

### 概述

`Memory` 是框架的消息存储器，核心职责是**存储 Message 并提供高效检索**。

### 数据结构

```go
type Memory struct {
    mu      sync.RWMutex          // 读写锁：读操作不互斥，写操作独占
    storage []*Message            // 有序消息列表（按插入顺序）
    index   map[string][]*Message // cause_by → 消息列表的倒排索引
    maxSize int                   // 容量上限，0 = 无限制
}
```

**双存储设计**：`storage` 保证时序，`index` 保证按 `cause_by` 的 O(1) 查找。

### 方法分类

| 类别 | 方法 | 复杂度 | 说明 |
|------|------|--------|------|
| **写入** | `Add` | O(1) | 追加 + 索引 + FIFO 淘汰 |
| **全量读取** | `Get(k)` | O(k) | 取最近 k 条 |
| | `Last` | O(1) | 取最后一条 |
| **索引读取** | `GetByAction` | O(1) | 按 cause_by 查 |
| | `GetByActions` | O(m) | 多 cause_by OR 查询 |
| **线性扫描** | `GetByRole` | O(n) | 按发送者遍历 |
| | `FindNews` | O(n) | 找未读消息 |
| **维护** | `Count` / `Clear` | O(1) / O(n) | |

### 关键设计点

**1. FIFO 淘汰（Add 方法）**

当 `len(storage) > maxSize` 时淘汰最旧消息。同时调用 `removeFromIndex` 保持索引和存储一致——否则索引中残留已删除消息的指针（**内存泄漏 + 脏数据**）。

**2. 为什么 GetByRole 用线性扫描而没有索引？**

`SentFrom`（发送者）基数通常很小（一个系统就几个 Role），建索引收益不大，反而增加写入开销和内存。`cause_by` 不同——每个 Action 都产生消息，种类多，索引价值高。

**3. 并发安全**

所有公开方法都用 `RLock`（读）或 `Lock`（写）保护。读写锁的收益：读取频率远高于写入时，多个 goroutine 可同时读而不互斥。

### 与 Message 的协作

```
Memory.storage → []*Message
                      ├── CauseBy  → Memory.index 的 key
                      ├── SentFrom → GetByRole 的过滤条件
                      ├── SendTo   → ShouldSendTo 的路由判断
                      └── ID       → FindNews 的去重依据
```

`Memory` 是"仓库"，`Message` 是"货物"，两者共同支撑起 `Environment` 的消息路由和 `Role` 的订阅机制。

### 消息已读/未读机制

> 涉及文件：[memory.go](../internal/foundation/memory.go) + [role.go](../internal/role/role.go)

**核心思想：不在 Message 上加 `Read` 字段，而是每个 Role 各自记账。**

```
Role.observed []string          ← 每个 Role 私有的已读 ID 列表
        ↓
markObserved(msg)               ← 收到消息就追加 ID
        ↓
Memory.FindNews(observed, k)    ← 差集运算：storage 中不在 observed 里的 = 未读
```

**Step 1 — Role 收到消息后标记已读（role.go:217-220）**

```go
r.memory.Add(msg)       // 写入 Memory 仓库
r.markObserved(msg)     // 记录 ID 到自己的 observed 列表
```

消息一旦通过 `shouldObserve` 过滤并被 Role 处理，就算"已读"。

**Step 2 — markObserved 追加 ID，硬上限 1000（role.go:417-422）**

```go
func (r *Role) markObserved(msg *foundation.Message) {
    r.observed = append(r.observed, msg.ID)
    if len(r.observed) > 1000 {
        r.observed = r.observed[100:] // 丢弃最旧的 100 条，保留最近 900 条
    }
}
```

1000 条是硬上限，防止长期运行的 Role 内存无限增长。超过时丢弃最旧的 100 条（意味着这些"已读"记录会被遗忘，对应的消息会重新变成"未读"——但通常这些消息早已被淘汰或不再需要）。

**Step 3 — FindNews 做差集（memory.go:111-123）**

```go
func (m *Memory) FindNews(observed []string, k int) []*Message {
    observedSet := make(map[string]bool, len(observed))  // 转 set，O(1) 查重
    for _, id := range observed {
        observedSet[id] = true
    }
    var news []*Message
    for i := len(m.storage) - 1; i >= 0 && len(news) < k; i-- {  // 从最新往前扫
        if !observedSet[m.storage[i].ID] {  // 不在已读集合 = 未读
            news = append(news, m.storage[i])
        }
    }
    return news
}
```

**设计优点：**

| 维度 | 说明 |
|------|------|
| **无状态污染** | Message 是纯数据对象，不嵌入运行时状态 |
| **多 Role 解耦** | 每个 Role 各自维护 `observed`，A 读了不代表 B 也读 |
| **无并发竞争** | `observed` 是 Role 私有字段，只在主循环 goroutine 中读写 |
| **低成本** | 差集运算只在调用 `FindNews` 时才发生，不是每条消息都做 |

**与"推送 vs 拉取"的关系：**

- `shouldObserve(watchKeys)` → **推送端过滤**：Environment 把消息投递到 msgBuffer 前，先判断该 Role 是否关注此 cause_by
- `FindNews(observed)` → **拉取端查询**：Role 主动查询"我还没看过哪些消息"
- `markObserved` → **状态标记**：处理完一条就记录一条

三者配合，形成完整的消息投递→过滤→标记→查询闭环。

# tool 包

## Tool 接口 + ToolRegistry

> 源文件：[internal/tool/tool.go](../internal/tool/tool.go)

### 概述

`tool.go` 定义了 Agent 框架的**工具层**——不依赖 LLM 的基础操作能力。如果把 Agent 比作一个人，Tool 是"手"（执行确定性操作），Action 是"脑"（LLM 驱动的决策）。

### Tool 与 Action 的分工

| | Tool | Action |
|---|---|---|
| 依赖 LLM？ | ❌ 不依赖 | ✅ AskLLM 是核心方法 |
| 本质 | 纯函数 | LLM 驱动的决策流程 |
| 示例 | ReadFile、WriteFile、RunCommand | WriteCode、ReviewCode |

**为什么要分开？** 如果文件读写和代码生成混在一起，LLM 调用会变得不可控。Tool 提供确定性的底层能力，Action 在此基础上做智能决策。

### 公有/私有分层

- **公有 Tool** 注册在 Environment 级别 → 所有 Role 共享（如 ReadFile，谁都需要）
- **私有 Tool** 注册在 Role 级别 → 只有该 Role 能用（如某个 Role 专属的外部 API）

### Tool 接口

```go
type Tool interface {
    Name() string                          // 唯一标识
    Description() string                   // 给 LLM 看的说明
    Parameters() map[string]string         // 参数说明
    Run(ctx context.Context, args map[string]any) (string, error)  // 执行
}
```

**设计核心**：`Description()` 和 `Parameters()` 的输出会注入 LLM 的 system prompt，让 LLM 知道有哪些工具可用以及如何调用。这意味着**工具描述即 prompt 工程**——描述写得好不好直接影响 LLM 调用工具的准确率。

### ToolRegistry 注册表

```go
type ToolRegistry struct {
    mu    sync.RWMutex       // 读写锁：读不互斥，写独占
    tools map[string]Tool    // name → Tool
}
```

| 方法 | 复杂度 | 说明 |
|------|--------|------|
| `Register(t)` | O(1) | 注册工具，重名 panic（视为编程错误） |
| `RegisterAll(tools...)` | O(n) | 批量注册 |
| `Get(name)` | O(1) | 按名查询，不存在返回 nil |
| `Call(ctx, name, args)` | O(1) | 查询 + 执行 |
| `List()` | O(n log n) | 按字母排序返回名称列表 |
| `Count()` | O(1) | 返回数量 |
| `Merge(other)` | O(n) | 合并另一个注册表，重名不覆盖 |
| `ToolsInfo()` | O(n log n) | 生成 LLM system prompt 用的工具描述文本 |

**关键设计点：**

1. **重名用 panic 而非 error** — 工具重名是编程错误，应在开发阶段暴露，panic 强制修复
2. **Merge 重名不覆盖** — 私有工具优先级高于公有工具。如果 Role 注册了自己的 `ReadFile`（加了权限检查），Environment 的通用版本不会覆盖它
3. **ToolsInfo 排序输出** — 保证每次生成的 prompt 顺序一致，对 LLM prompt cache 友好
4. **空注册表返回空字符串** — 不出现空 "Available Tools" 段落，避免 LLM 困惑

### Merge 的设计意图

```
Environment.ToolRegistry (公有) ──Merge──→ Role.ToolRegistry (私有)
                                              ├── 已有私有工具（保留）
                                              └── 公有工具（追加，不覆盖同名私有工具）
```

### System Prompt 生成

`ToolsInfo()` 输出格式：
```
## Available Tools
1. **ReadFile**: 读取指定路径的文件内容
   Parameters: path (文件路径)
2. **WriteFile**: 将内容写入文件
   Parameters: path (文件路径), content (文件内容)
```

### 与 LLM 的协作流程

```
初始化阶段:
  注册所有 Tool → ToolsInfo() 生成描述 → 注入 LLM system prompt

运行阶段:
  LLM 输出"调用 ReadFile path=xxx" → Call(ctx, "ReadFile", args) → Tool 执行 → 结果返回 LLM
```

### 整体架构图

```
┌─────────────────────────────────────────────┐
│                 Environment                  │
│  ┌───────────────────────────────────────┐  │
│  │       ToolRegistry (公有)              │  │
│  │  ReadFile │ WriteFile │ RunCommand    │  │
│  └───────────────────────────────────────┘  │
│                    │                         │
│              Merge 注入                      │
│         ┌─────────┼─────────┐               │
│         ▼         ▼         ▼               │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐    │
│  │  Role A  │ │  Role B  │ │  Role C  │    │
│  │ 私有工具  │ │ 私有工具  │ │ 私有工具  │    │
│  │+公有工具  │ │+公有工具  │ │+公有工具  │    │
│  └──────────┘ └──────────┘ └──────────┘    │
│                                              │
│  ToolsInfo() → 注入 LLM System Prompt        │
│  LLM 看到可用工具列表 → 决策调用哪个工具       │
└─────────────────────────────────────────────┘
```

| `context.Context` | 第 42 行 | 传递超时/取消/截止时间信号 |

### 与 builtin 工具的关系

`tool.go` 只定义接口和注册表，具体实现在 `internal/tool/builtin/` 目录：

| 文件 | 工具 | 说明 |
|------|------|------|
| [read_file.go](../internal/tool/builtin/read_file.go) | ReadFileTool | 读文件，限制输出 10000 字符 |
| [write_file.go](../internal/tool/builtin/write_file.go) | WriteFileTool | 写文件 |
| [run_command.go](../internal/tool/builtin/run_command.go) | RunCommandTool | 执行命令 |

每个 builtin 工具都是一个空 struct（如 `type ReadFileTool struct{}`），实现 `Tool` 接口的 4 个方法。空 struct 不占用内存，是 Go 中"无状态服务"的惯用写法。
