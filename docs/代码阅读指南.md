# Cohort 代码阅读指南

> 框架已有 24 个 Go 源文件、9 个包、~3,500 行代码。按依赖关系从底向上读，每层 10-15 分钟。

---

## 阅读顺序总览

```
第 1 步: foundation/     ← 地基（20 分钟）
第 2 步: tool/           ← 工具（10 分钟）
第 3 步: llm/            ← LLM 调用（25 分钟）
第 4 步: action/         ← 动作（15 分钟）
第 5 步: role/           ← ★ 心脏（25 分钟）
第 6 步: env/            ← 路由（15 分钟）
第 7 步: team/           ← 编排（10 分钟）
第 8 步: cmd/ demos      ← 串联（20 分钟）
```

**为什么这顺序？** 框架的依赖关系是自底向上的：

```
foundation ← tool ← llm ← action ← role ← env ← team ← cmd
```

后面的包 import 前面的包。按这顺序读，永远不遇到还没见过的类型。

## 核心数据流（边读边对照）

```
用户需求: "写一个 2048 游戏"
  │
  ▼
Team.Run(ctx, idea)
  │ NewUserMessage(idea)
  ▼
Environment.PublishMessage(msg)         ← [env] 消息进入全局历史
  │ 遍历 Role, msg.ShouldSendTo(name)?
  ▼
Role.msgBuffer ← msg                    ← [channel] 非阻塞推入
  │
  ▼
Role.Run() 主循环 select 收到 msg       ← [goroutine] 独立执行
  │
  ├→ shouldObserve(msg)?               ← watch 匹配
  ├→ memory.Add(msg)                   ← 存入 Memory
  ├→ injectToolsToAction(act)          ← 注入公有+私有 Tool
  ├→ react(ctx)                        ← context.WithoutCancel 保护
  │   └→ act.Run(ctx, history)
  │       └→ AskLLM(prompt, history)
  │           ├→ system prompt + Tool 列表
  │           ├→ 转 history → ChatMessage[]
  │           └→ client.Chat(messages)  ← [llm] 调 LLM
  │       └→ CallTool("WriteFile", ...) ← [tool] 写磁盘
  │       └→ CallTool("RunCommand", ...)← [tool] 编译
  └→ env.PublishMessage(rsp)           ← 发布结果，下一个人收到
```

## 读代码的 3 个建议

1. **看接口，别看实现**：每打开一个包，先找 `type XXX interface`。接口就是这层的"契约"。
2. **跟着一条消息走**：从 `NewUserMessage` → `PublishMessage` → `msgBuffer` → `Run()` → `react()` → `PublishMessage`，把整条链路追踪一遍。
3. **改 demo 验证理解**：改一下 `demo_duo` 的 task，加一个 Role，换一个 Action，看能不能跑通。
