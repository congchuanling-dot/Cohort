## 1. 第一阶段：基础设施层

> **原则**：纯标准库，零外部依赖。

### 文件清单

| 文件 | 说明 |
|------|------|
| [config.go](../internal/foundation/config.go) | 全局配置：LLMConfig + AgentConfig + WorkspaceConfig，三级配置覆盖结构 |
| [message.go](../internal/foundation/message.go) | 通信单元：ID / Content / CauseBy / SentFrom / SendTo 路由字段 |
| [memory.go](../internal/foundation/memory.go) | 消息存储：FIFO 淘汰 + cause_by 索引 + 多维度查询 |
| [logger.go](../internal/foundation/logger.go) | 结构化日志：基于 `log/slog`，支持 Info/Debug 切换 |

### 核心类型

```
Config
  ├── LLMConfig      ← LLM 配置单元（Provider + Model + 参数）
  ├── RolesLLMConfig  ← 按 Role 名称覆盖（第 2 层优先级）
  ├── ActionsLLMConfig ← 按 Action 名称覆盖（第 3 层，最高优先级）
  ├── AgentConfig     ← Agent 运行参数
  └── WorkspaceConfig ← 工作区配置

Message
  ├── CauseBy         ← 由哪个 Action 产生（Role 订阅依据）
  ├── SentFrom        ← 由哪个 Role 发送
  ├── SendTo          ← 接收者列表（RouteToAll = 广播）
  └── ShouldSendTo / ShouldObserve  ← 路由和过滤判断

Memory
  ├── Add(msg)        ← 并发安全添加，FIFO 淘汰
  ├── Get(k)          ← 获取最近 k 条
  ├── GetByAction     ← 按 cause_by 索引查找
  ├── GetByRole       ← 按发送者查找
  └── FindNews        ← 拉取未读消息
```

```powershell
go run ./cmd/demo/
```

## 2. 第二阶段：LLM 调用层

> **核心目标**：用户通过 YAML 配置自由切换 LLM 提供商，框架内部对上层完全透明。
> **设计原则**：面向接口编程 + 适配器模式。

### Batch 1：Client 接口 + 内部类型

| 文件 | 说明 |
|------|------|
| [types.go](../internal/llm/types.go) | Provider 类型枚举（openai/deepseek/anthropic/ollama/custom） |
| [client.go](../internal/llm/client.go) | Client 接口 + ChatMessage / ChatResponse / TokenUsage / StreamChunk |

**Client 接口**——上层唯一依赖：

```go
type Client interface {
    Chat(ctx, []ChatMessage) (*ChatResponse, error)
    ChatStream(ctx, []ChatMessage) (<-chan *StreamChunk, error)
    CountTokens([]ChatMessage) int
    Name() string
}
```

4 个内部类型（框架自己的抽象，不和任何具体 API 耦合）：

```
ChatMessage   ← 对话消息（Role + Content）
ChatResponse  ← LLM 响应（Content + FinishReason + Usage）
TokenUsage    ← Token 统计
StreamChunk   ← 流式片段（Content + Done + Error）
```

### Batch 2：Provider 注册工厂 + 三级配置继承

| 文件 | 操作 | 说明 |
|------|------|------|
| [registry.go](../internal/llm/registry.go) | 新建 | ProviderConfig + ProviderFactory + Register/NewClient + LLMResolver |
| [config.go](../internal/foundation/config.go) | 修改 | 新增 RolesLLMConfig / ActionsLLMConfig + Config 新增 Roles/Actions 字段 |

**LLMResolver —— 三级配置继承**：

```
Resolve(roleName, actionName)
  ├→ 1. 从全局默认开始
  ├→ 2. 如果 roles.{roleName} 有覆盖 → merge
  └→ 3. 如果 actions.{actionName} 有覆盖 → merge（最高优先级）

示例：
  Resolve("Alex", "WriteCodeReview")
    → actions.WriteCodeReview 覆盖 → Claude Haiku（Action 赢了 Role 的 Opus）
  Resolve("Edward", "WriteTest")
    → roles.Edward 覆盖 → DeepSeek
  Resolve("Alice", "WritePRD")
    → 无覆盖 → 全局默认 deepseek-chat
```

### Batch 3：OpenAI Provider

| 文件 | 说明 |
|------|------|
| [provider_openai.go](../internal/llm/provider_openai.go) | OpenAI 独立实现（~200 行） |

核心设计：**双向翻译**

```
上层代码（Action/Role）
    │ 只依赖 ChatMessage / ChatResponse
    ▼
┌─────────────────────────────────────────┐
│         openaiClient.Chat()             │
│  1. ChatMessage[] → openaiMessage[]     │  框架格式 → OpenAI 格式
│  2. HTTP POST /chat/completions         │
│  3. OpenAI JSON → ChatResponse          │  OpenAI 格式 → 框架格式
└─────────────────────────────────────────┘
```

ChatStream：后台 goroutine 解析 SSE 流，返回 `<-chan *StreamChunk`。

| 文件 | 说明 |

**Anthropic 适配器的 4 个适配点**：

```
框架 ChatMessage              Anthropic Messages API
────────────────────────────────────────────────────
system 在 messages 数组里  →  system 提升为请求顶层字段
content 是纯字符串          →  content 是 [{type:"text", text:"..."}]
Authorization: Bearer xxx   →  x-api-key: xxx
POST /chat/completions      →  POST /v1/messages + anthropic-version 头
```

### 使用示例

```go
// 5 行代码创建客户端
client, _ := llm.NewClient("deepseek", llm.ProviderConfig{
    Model:  "deepseek-v4-pro",
    APIKey: os.Getenv("DEEPSEEK_API_KEY"),
})

// 同步对话
resp, _ := client.Chat(ctx, messages)

// 流式输出

for chunk := range ch {
    fmt.Print(chunk.Content)
}
```

## 3. 第三阶段：Agent 核心层

> **这是整个项目最核心的模块。面试时 70% 的问题集中在这里。**

### Batch 1：Action 接口 + 结构化输出解析

| 文件 | 说明 |
|------|------|
| [action.go](../internal/action/action.go) | Action 接口 + ActionOutput + BaseAction（~120 行） |
| [node.go](../internal/action/node.go) | ActionNode 结构化输出解析器（~95 行） |

**三个核心类型**：

```
Action (接口)          ← 每个具体动作都要实现
  ├── Name() string
  └── Run(ctx, history) → (*ActionOutput, error)

BaseAction (结构体)    ← 嵌入到具体 Action 里，提供 LLM 调用能力
  ├── AskLLM(ctx, prompt, history) → (string, error)    ★ 核心方法
  ├── AskLLMStream(ctx, prompt, history) → (<-chan, error)
  └── frameToLLMMessages(history) → []ChatMessage         ← 框架 → LLM 格式转换

ActionNode (结构体)    ← 从 LLM 非结构化输出中提取 JSON / 字段
  └── Fill(llmOutput, &target) → error
```

**AskLLM 调用链**：

```
子 Action.Run()
  └→ BaseAction.AskLLM(ctx, "请写一个 PRD...", history)
       │
       ├→ 1. 构建 [system: prefix]           ← system prompt
       ├→ 2. 追加 history（Memory 消息历史）   ← 上下文窗口
       ├→ 3. 追加 [user: "请写一个 PRD..."]   ← 当前任务
       └→ 4. client.Chat(ctx, messages)      ← 调 LLM
```

**ActionNode 两层策略**：

```
LLM 输出: "好的，这是 PRD...\n```json\n{\"title\":\"2048\"}\n```"
                                    ↓
Fill() → 策略 1: 提取 ```json 块 → json.Unmarshal ✅
         （失败则）策略 2: 按 Schema 正则逐字段提取
```

### Batch 2：Role 核心——整个框架的心脏

| 文件 | 说明 |
|------|------|
| [role.go](../internal/role/role.go) | Role 核心 + 3 种 React 模式（~400 行） |
| [context.go](../internal/role/context.go) | RoleContext 状态管理（~55 行） |

**Role 结构的三个层次**：

```
Role 结构体
  ├── 身份信息（Name, Profile, Goal, Constraints, Desc）
  │     ↑ 给 LLM 的 system prompt 用，设定人设
  │
  ├── 行为定义（actions[], reactMode, watch）
  │     ↑ 决定"能做什么"、"怎么做"、"关注什么"
  │
  └── 运行时（state, msgBuffer(channel), memory, env, observed）
        ↑ goroutine + channel 构成的并发执行单元
```

**核心循环（面试最重要的代码）**：

```
Run(ctx) ← 在 goroutine 中运行
  │
  └→ for { select {
       case <-ctx.Done():    → 优雅退出
       case msg := <-msgBuffer:  → 收到消息
         │
         ├→ Step 1: Observe  → shouldObserve(msg) ? → memory.Add + markObserved
         ├→ Step 2+3: React  → reactByOrder / reactReAct / reactPlanAndAct
         └→ Step 4: Publish  → env.PublishMessage(rsp)
     }}
```

**三种 React 模式**：

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| `ReactByOrder` | 按 Actions 列表顺序逐个执行 | 软件公司 SOP（PM→Architect→Engineer→QA） |
| `ReactReAct` | LLM 根据当前状态动态选择 Action | 开放域任务，不确定下一步做什么 |
| `ReactPlanAndAct` | 先规划步骤列表，再执行 | 复杂多步骤任务 |

**MessagePublisher 接口——解耦 Environment**：

```go
type MessagePublisher interface {
    PublishMessage(msg *foundation.Message)
}
```

## 4. 第四阶段：环境与编排层

### Environment + Team

| 文件 | 说明 |
|------|------|
| [environment.go](../internal/env/environment.go) | 消息路由中心 + Role 生命周期管理（~190 行） |
| [team.go](../internal/team/team.go) | 多 Agent 团队编排器（~120 行） |

**Environment —— 消息路由中心**：

```
Environment
  ├── RegisterRole(r, addresses...)   ← 注册 Role + 建立双向连接
  ├── PublishMessage(msg)             ← 路由消息到匹配的 Role（非阻塞）
  ├── Run(ctx)                        ← 并发运行所有活跃 Role + 轮询等待
  ├── IsAllIdle()                     ← 检查全员空闲
  └── History()                       ← 全局消息历史
```

**路由机制**：

```
PublishMessage(msg)
  → 遍历所有 Role
  → msg.ShouldSendTo(name) 匹配？
  → 匹配则非阻塞推入 msgBuffer
```

**Team —— 编排器**：

```
Team.Run(ctx, "写一个 2048 游戏")
  │
  ├→ 1. PublishMessage(用户需求)     ← 广播给所有 Role
  │
  ├→ 2. for round = 1..nRound:
  │       env.Run(ctx)                ← 并发执行所有活跃 Role
  │       if IsAllIdle() → break      ← 全员完成，提前收工
  │
  └→ 3. env.Archive()                ← 归档结果
```

### 内置 Action

**WritePRD 的 Run 模式**：

```go
func (a *WritePRD) Run(ctx, history) (*ActionOutput, error) {
    userReq := extractUserRequirement(history)  // 从历史中找到用户需求
    content, _ := a.AskLLM(ctx, "Please write PRD...\n" + userReq, history)
    return &ActionOutput{Content: content}, nil
}
```

### 双 Agent 协作 Demo

**协作流程**：

```
User（"写一个网页版 2048 游戏的需求文档"）
  │
  ├→ Alice (PM, watch=UserRequirement, action=WritePRD)
  │     │ 观察到 UserRequirement
  │     │ 执行 WritePRD → 调用 DeepSeek → 生成 PRD
  │     └→ 发布 WritePRD
  │
  └→ Bob (Reviewer, watch=WritePRD, action=WriteCodeReview)
        │ 观察到 WritePRD
        │ 执行 WriteCodeReview → 调用 DeepSeek → 生成评审意见
        └→ 发布 WriteCodeReview
```

**实际运行结果**（30 秒完成）：

```
20:40:34 Alice 启动，观察到 UserRequirement，执行 WritePRD
20:40:50 Alice 完成 → 发布 PRD（3,458 字，16 秒 LLM 调用）
20:40:50 Bob 观察到 WritePRD，执行 WriteCodeReview
20:41:04 Bob 完成 → 发布评审意见（3,070 字，14 秒 LLM 调用）
20:41:05 全员空闲，第 1 轮完成 ✅
```

### 关键修复

| 问题 | 修复 |
|------|------|
| env.Run() 死等 goroutine | 改为轮询 IsAllIdle + 超时 cancel |
| LLM 调用被 ctx 取消打断 | `context.WithoutCancel(ctx)` 保护 LLM 调用 |
| DEEPSEEK_API_URL 双重路径 404 | `trimChatPath()` 自动截掉 `/v1/chat/completions` |
| SSE 流式空白 | 兼容 `data:`/`data: ` 前缀 + 自然结束补发 Done |

### Tool 层与公有工具注册

| 文件 | 说明 |
|------|------|
| [tool/tool.go](../internal/tool/tool.go) | **核心**：Tool 接口 + ToolRegistry（注册/调用/ToolsInfo） |
| [tool/builtin/read_file.go](../internal/tool/builtin/read_file.go) | 公有 Tool：读取文件 |
| [tool/builtin/write_file.go](../internal/tool/builtin/write_file.go) | 公有 Tool：写入文件（自动创建目录） |
| [tool/builtin/run_command.go](../internal/tool/builtin/run_command.go) | 公有 Tool：执行系统命令（30s 超时） |

**Tool 接口**：

```go
type Tool interface {
    Name() string                       // 唯一标识
    Description() string                // 给 LLM 看的描述
    Parameters() map[string]string      // 参数说明
    Run(ctx, args) (string, error)      // 执行
}
```

**公有/私有工具架构**：

```
Environment.publicTools     ← 公有（WriteFile / ReadFile / RunCommand）
    ↓ RegisterRole 时自动注入
Role.tools                  ← 合并 = 公有 + 私有
    ↓ injectToolsToAction
Action.tools                ← Action.Run() 时可用
    ↓ AskLLM 自动注入
system prompt               ← LLM 看到完整可用工具列表
```

**之前 vs 之后**：

```go
// 之前：每个 Role 手动配工具（重复）
charlie := role.NewRole("Charlie",
    role.WithTools(tools.NewWriteFileTool(), tools.NewRunCommandTool()),
)

// 之后：Environment 注册一次，全员自动继承
env.RegisterPublicTool(tools.NewWriteFileTool())
env.RegisterPublicTool(tools.NewRunCommandTool())
charlie := role.NewRole("Charlie", ...) // 自动拥有

// 私有工具照常单独配
charlie := role.NewRole("Charlie",
    role.WithTools(goFmtTool), // ← 只有 Charlie 能用
)
```

**Tool 的 LLM 集成**：

AskLLM 自动将可用工具列表注入 system prompt：

```
## Available Tools
1. **WriteFile**: 将内容写入指定路径的文件...
   Parameters: path (文件路径), content (要写入的文件内容)
2. **RunCommand**: 执行一个系统命令并返回输出...
   Parameters: command (要执行的命令), workdir (工作目录)
```

**WriteCode 集成 Tool**：

LLM 返回代码后，Action 自动提取代码块 → 调 WriteFile 写磁盘 → 调 RunCommand 编译：

```go
func (a *WriteCode) Run(ctx, history) {
    content, _ := a.AskLLM(ctx, prompt, history)     // LLM 生成代码
    toolResults := a.saveAndBuild(ctx, content)       // ★ 调 Tool 写文件+编译
    return &ActionOutput{Content: content + toolResults}
}
```

## 8. 关键设计决策

### 8.1 面向接口编程 + 适配器模式

上层代码（Action/Role）只依赖 `Client` 接口和 `ChatMessage`/`ChatResponse` 内部类型。

切 LLM 提供商**只改 YAML，Go 代码零改动**。

### 8.2 三级配置继承

```
Action 级别（最高）  →  如 WriteCode 强制用 Claude
  ↓ 继承
Role 级别（中等）    →  如 Alex（Engineer）整体用 Claude
  ↓ 继承
全局默认（兜底）     →  其余用 deepseek-chat
```

只填想覆盖的字段，其余自动继承（字段级 merge）。

### 8.3 Provider 注册表

每个 Provider 在 `init()` 中自我注册：

```go
func init() {
    Register("deepseek", newDeepSeek)
}
```

`import _ "..."` 即可接入新 Provider，无需修改其他代码。

### 8.4 context.WithoutCancel 保护 LLM 调用

env 的轮询机制会在超时后 cancel context 来停止 goroutine，但正在进行的 LLM 调用不能被中断（否则浪费 token 和费用）。

`context.WithoutCancel(ctx)` 确保 LLM 调用一旦发起就必须完成。

### 8.5 非阻塞消息投递

```go
select {
case r.MessageBuffer() <- msg:
    // 成功
default:
    // buffer 满 → 丢弃并告警，不阻塞发送者
}
```

防止慢消费者拖死整个系统。

### 8.6 组合复用替代继承

DeepSeek 和 Ollama 通过嵌入 `*openaiClient` 复用所有逻辑：

```go
type deepseekClient struct {
    *openaiClient  // ← 零额外代码
}
// 只覆盖 Name() 和默认 BaseURL
```

56 行代码实现一个完整 Provider。

### 8.7 MessagePublisher 接口解耦

Role 不依赖完整的 Environment，只依赖 `MessagePublisher` 接口：

```go
type MessagePublisher interface {
    PublishMessage(msg *foundation.Message)
}
```

这使 Role 可以独立编译和测试，也为未来替换消息中间件留了空间。

### 8.8 函数式选项模式

```go
alice := role.NewRole("Alice",
    role.WithProfile("PM", "Write clear PRDs", "Be concise"),
    role.WithActions(writePRD),
    role.WithWatch("UserRequirement"),
    role.WithMemory(mem),
)
```

比构造函数参数列表更清晰，同时保持零值安全的默认行为。

### 8.9 公有 Tool 注册 + 私有 Tool 隔离

**Action = LLM 驱动的复杂行为（"做什么"），Tool = 不依赖 LLM 的基础能力（"用什么做"）。**

公有 Tool 注册在 Environment 级别，所有 Role 自动继承（只写一次）：

```go
env.RegisterPublicTool(tools.NewWriteFileTool())
env.RegisterPublicTool(tools.NewRunCommandTool())
// 之后 Hire 的所有 Role 都自动拥有以上工具
```

私有 Tool 注册在 Role 级别，只有该 Role 能用：

```go
alice := role.NewRole("PM",
    role.WithTools(stakeholderMapTool), // ← 只有 PM 用
)
```

Tool 列表自动注入到 LLM 的 system prompt，让 LLM 知道有哪些工具可用、何时调用。Action 在 LLM 返回后通过 `CallTool()` 执行实际操作（写文件、跑命令等）。

### 8.10 组合复用替代继承

DeepSeek 和 Ollama 通过嵌入 `*openaiClient` 复用所有逻辑：

```go
type deepseekClient struct {
    *openaiClient  // ← Chat/ChatStream/CountTokens 全部继承，零额外代码
}
// 只覆盖 Name() 和默认 BaseURL → 48 行完成一个完整 Provider
```
