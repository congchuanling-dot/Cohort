# Cohort vs MetaGPT 功能对比与后续开发计划

> 对照 MetaGPT 框架，分析 Cohort 已有能力和缺失功能，规划后续开发优先级。

---

## 一、Cohort 已有能力 ✅

| 模块 | 覆盖内容 | 状态 |
|------|---------|------|
| **Foundation** | Config（三级配置结构）、Message（CauseBy/SendTo 路由）、Memory（FIFO + 多维度索引）、Logger（slog） | ✅ |
| **LLM** | Client 接口 + 5 Providers（OpenAI/DeepSeek/Anthropic/Ollama/Custom）+ 三级配置继承 + Registry + Mock | ✅ |
| **Action** | Action 接口 + BaseAction + ActionOutput + ActionNode | ✅ |
| **内置 Action** | WritePRD、WriteDesign、WriteCode、WriteCodeReview（4 个） | ✅ |
| **Role** | observe→think→act 循环 + 3 种 React 模式（ByOrder/ReAct/PlanAndAct）+ RoleContext | ✅ |
| **Environment** | 消息路由中心 + 非阻塞投递 + 公有 Tool 注册 | ✅ |
| **Team** | Hire + Run + 预算/轮次控制 | ✅ |
| **Tool** | Tool 接口 + ToolRegistry + 公有/私有隔离 + 3 个内置 Tool（ReadFile/WriteFile/RunCommand） | ✅ |
| **Demo** | Foundation 演示、LLM 层演示、双 Agent 协作、3 Agent 软件公司、Tool 写盘编译（5 个） | ✅ |
| **文档** | README + 开发过程总结 + 开发路线图（3 份） | ✅ |

### 架构全景

```
cmd/ (5 demos)
  └→ team/        Team.Hire + Run
       └→ env/    Environment（消息路由 + 公有 Tool 注册）
            └→ role/  Role（observe→think→act）
                 └→ action/  Action（WritePRD/WriteCode/...）
                      ├→ AskLLM → llm/  Client（OpenAI/DeepSeek/...）
                      └→ CallTool → tool/  Tool（ReadFile/WriteFile/RunCommand）
                           └→ foundation/  Config/Message/Memory/Logger
```

## 二、Cohort 缺失功能清单

### P0 —— 阻碍体验的关键缺失

| 序号 | 功能 | MetaGPT 实现 | 说明 | 预估 |
|------|------|-------------|------|------|
| 1 | **YAML 配置加载** | `Config.from_yaml()` + 环境变量覆盖 | 现在启动 Demo 要写一堆 Go 代码配 Config。有了 YAML，一个配置文件 + `foundation.Load("config.yaml")` 搞定 | 30分钟 |
| 2 | **Planner / Plan** | `Planner` + `Plan` + `Task` + 计划-执行追踪 | PlanAndAct 模式现在是空的，直接回退到 ByOrder。这是 Agent 框架和"调一次 LLM"的本质区别 | 1-2天 |
| 3 | **Planner Prompt + 计划状态管理** | `ThinkingCommand`（APPEND/RESET/REPLACE/FINISH）+ `plan_status` 渲染 | 让 LLM 能动态修改计划，而非一次性生成 | 包含在 Planner 里 |

### P1 —— 完善框架能力

| 序号 | 功能 | MetaGPT 实现 | 说明 | 预估 |
|------|------|-------------|------|------|
| 4 | **更多内置 Action** | 20+ 个（WriteTasks/RunCode/DebugError/FixBug/SearchWeb/WriteTest 等） | 现在只有 4 个，覆盖场景太少。至少加 WriteTasks、WriteTest、RunCode、SearchWeb | 每个 15分钟 |
| 5 | **更多内置 Role** | 20+ 个（PM/Architect/Engineer/QA/Researcher/RoleZero 等） | Cohort 没有预设 Role，全靠手动 WithActions + WithWatch 搭配。加 NewPM()/NewEngineer() 等工厂函数 | 每个 10分钟 |
| 6 | **Code Sandbox / Workspace 隔离** | `Workspace` + UID 隔离目录 + `GitRepository` | WriteFileTool 现在写到项目根目录，污染代码。封装 Workspace 管理器，统一输出到隔离目录 | 30分钟 |
| 7 | **Token 压缩** | `Compressor` 4 种策略（PostCutByToken 等） | 开发路线写了 `compress.go` 但没实现。Memory 取 20 条消息可能超过 LLM 上下文限制 | 30分钟 |
| 8 | **重试机制** | 指数退避重试 | 开发路线写了 `retry.go` 但没实现。网络抖动时 LLM 调用直接失败 | 20分钟 |

### P2 —— 增强工程化

| 序号 | 功能 | MetaGPT 实现 | 说明 | 预估 |
|------|------|-------------|------|------|
| 9 | **预算 / Cost 追踪** | `CostManager` + 模型价格表 + 预算超支抛异常 | Team 有 budget 字段但没实现检查逻辑 | 30分钟 |
| 10 | **更多 Tool** | SearchWeb、Git、Browser、Editor、Deployer、CodeReview 等 18+ 个 | 现在只有 3 个基础 Tool。至少加 SearchWeb | 每个 20分钟 |
| 11 | **序列化 / 断点恢复** | `Team.serialize()` + `Team.deserialize()` | 协作跑一半出错了，无法从断点恢复，要重头来 | 1天 |
| 12 | **角色恢复** | `recovered` 标志 + `latest_observed_msg` | Role 中断后重启时能继续上次的状态 | 包含在上一条 |
| 13 | **人机交互** | `ask_human` / `reply_to_human` / `HumanProvider` | 不能让用户中途介入、审批或提修改意见 | 40分钟 |
| 14 | **消息压缩** | `BrainMemory.summarize()` | 长期对话时自动摘要旧消息，而不是直接丢弃 | 40分钟 |

### P3 —— 高级特性

| 序号 | 功能 | MetaGPT 实现 | 说明 |
|------|------|-------------|------|
| 15 | **RAG / 向量检索** | 完整 RAG 引擎 + BM25/FAISS/Chroma 多种后端 | 让 Agent 能检索外部知识库 |
| 16 | **经验池** | `@exp_cache` 装饰器 + ExperienceManager | 缓存 Action 结果，避免重复调 LLM |
| 17 | **长期记忆** | FAISS 向量存储 + 超容量自动转入 | 超过 Memory 容量时转入长期存储 |
| 18 | **流式 UI** | `LLM_STREAM_QUEUE` | StreamChunk 已有，但没有消费端展示 |
| 19 | **多语言支持** | `language` 配置贯穿所有 Role/Action | 统一控制输出语言 |
| 20 | **自动归档** | `GitRepository.archive()` | 自动 git commit 所有产出物 |
| 21 | **并发 Web 研究** | `asyncio.gather` 并发搜索 | 多个搜索任务并行执行 |
| 22 | **树状思维（TOT）** | `ThoughtNode` + `ThoughtTree` + 多种搜索策略 | 复杂推理任务的搜索空间探索 |
| 23 | **ActionGraph** | 有向图表示 Action 依赖 + 拓扑排序 | 并行 Action 依赖管理 |
| 24 | **扩展场景** | 狼人杀 / StanfordTown / 狼人杀 | 非软件开发场景的多 Agent 游戏/模拟 |

## 三、建议开发路线

### 近期（本周可完成）

```
1. YAML 配置加载         ← 30 分钟，体验质变
2. 重试机制               ← 20 分钟，网络抖动不再致命
3. Token 压缩             ← 30 分钟，长对话不超上下文
4. Workspace 沙箱         ← 30 分钟，不污染项目目录
5. 预算追踪               ← 30 分钟，控制成本
```

### 中期（下周）

```
6. Planner + Plan         ← 核心，让 Agent 真正做多步骤任务
7. 更多内置 Action/Role    ← 覆盖常见场景
8. 更多 Tool              ← SearchWeb / Editor
```

### 远期（按需）

```
9. 序列化 / 断点恢复       ← 长任务场景
10. RAG / 向量检索         ← 知识密集型场景
11. 经验池 / 长期记忆      ← 复用性高
12. TOT / ActionGraph     ← 复杂推理
```

## 五、核心竞争力总结

### 面试时已经可以说的话

1. **"从零实现了多 Agent 框架，纯 Go 标准库，零外部依赖"**
2. **"每个 Agent 是独立的 goroutine + channel，消息驱动并发执行"**
3. **"5 个 LLM Provider 可插拔，适配器模式处理 API 格式差异，三级配置继承"**
4. **"Action 和 Tool 两层分离——Action 调 LLM 决策，Tool 执行实际 IO"**
5. **"消息路由用 CauseBy + SendTo 控制流向，非阻塞投递防止慢 Agent 拖死系统"**
6. **"3 Agent 软件公司协作：PM → Architect → Engineer，85 秒产出完整可用代码"**

### 加完 P0/P1 后可以说的

7. **"Planner 支持动态规划：Agent 先生成计划，再逐步执行和追踪进度"**
8. **"YAML 配置驱动：切换 LLM、调整 Agent 行为全在配置文件中完成，不改代码"**
9. **"Token 压缩 + 指数退避重试 + 预算追踪——生产级 LLM 调用的三大保障"**
